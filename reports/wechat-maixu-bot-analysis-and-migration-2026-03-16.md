# 一、执行摘要

这份代码不是简单的“排麦脚本”，而是一套偏成熟的**微信语音厅麦序机器人主控程序**。它的核心结构是：

- **个人微信 Hook / xBot 作为接入层**
- **本地 Python 单体程序作为调度与状态机核心**
- **远程 RoomApi 服务作为配置中心、统计中心、权益中心**
- **群聊消息 + 定时任务共同驱动业务流程**

它已经具备完整的运营化能力，包括：

- 按小时自动开档 / 截档
- 关键词扣排、手速榜、最终榜、补排
- 买8、插队、取排、作废、麦序转、手动设榜
- 主持排班、打卡记录、日报周报月报、Excel 导出
- 置顶卡、黑名单、累计任务、VS 打榜
- 报备回厅提醒
- 多种防脚本 / 防扣排器机制
- 房间远程配置、激活码续期、自动更新程序

**优点**：业务理解深、功能全、配置能力强、非常贴近微信群运营场景。  
**缺点**：底座依赖个人微信 Hook，单文件巨石，状态强耦合，并发与维护风险较高。  

如果从长期产品化角度看，这套系统最值得保留的是：

1. **按小时档驱动的业务状态机**
2. **关键词 → 数值 value 的排榜模型**
3. **房间配置中心 + 统计中心的产品抽象**
4. **主持排班 / 打卡 / 报表 / 权益系统这些业务资产**

最应该替换的是：

1. **个人微信 Hook 接入方式**
2. **单体 Python 文件 + 全局内存状态**
3. **群聊天文本承担全部交互**

# 二、代码整体定位

## 2.1 一句话定义

这是一个基于**个人微信号 Hook 框架**运行的**语音厅麦序机器人主程序**，负责：

- 接收微信群消息
- 在特定时间发出开档信息
- 接收扣排内容并实时更新麦序榜
- 在档期结束后发最终榜与统计
- 把数据同步到远程服务
- 提供大量管理员与运营功能

## 2.2 产品形态判断

从能力上看，它更像：

> 一个“微信群语音厅麦序 SaaS 的本地执行端”

不是简单机器人，而是：

- 微信接入代理
- 本地业务执行引擎
- 远程配置执行器
- 运营统计采集器

# 三、系统架构拆解

可以粗分为 5 层。

## 3.1 微信接入层（x_Bot）

通过 `x_Bot` 与个人微信 Hook 框架交互，能力包括：

- 获取当前微信信息
- 获取群列表
- 获取群成员列表
- 收群消息（通过 webhook 回调）
- 发群消息
- @成员
- 发表情、图片、文件、卡片
- 修改群昵称 / 群公告
- 自动进群 / 退群

这一层是典型的**非官方微信适配层**。

## 3.2 本地 HTTP 回调层

程序使用：

- Flask
- gevent WSGIServer
- `POST /`

接收来自 xBot 的消息事件，作为整个程序的事件入口。

## 3.3 本地运行状态层

所有房间的运行时状态都集中保存在：

- `all_configs_arr`

这个字典里既有：

- 配置缓存
- 运行状态
- 当前榜单
- 上一档榜单
- 主持排班缓存
- 报备状态
- 黑名单
- 打卡缓存
- 防脚本临时验证码
- 各种阶段开关

它本质上是一个**内存数据库**。

## 3.4 远程业务服务层（RoomApi）

通过 `RoomApi` 调远程接口，负责：

- 房间配置读取
- 房间配置更新
- 麦序统计上传
- 打卡记录上传
- 黑麦记录
- 历史数据查询
- 置顶卡查询/使用
- 激活码续期
- 创建默认房间
- 创建 AI 房间
- 导出相关数据

说明这套系统并不是纯本地模式，而是**本地执行 + 远程配置中心**架构。

## 3.5 定时调度层

用了多个调度器：

- `scheduler`：房间开档任务
- `schedulerMain`：全局周期检查任务
- `schedulerBBTime`：报备提醒任务
- `schedulerUpload`：延迟上传 / 日报任务

说明程序是一个**事件驱动 + 时间驱动**的混合系统。

# 五、启动与运行流程

程序启动时会执行：

1. 注册退出钩子（中断/关闭时保存状态）
2. 从 `all_configs_arr.pkl` 恢复本地状态
3. 读取公共配置 `common_config`
4. 初始化远程 API：`RoomApi`
5. 初始化微信接入：`x_Bot`
6. 获取当前微信号信息
7. 获取当前已加入群列表
8. 拉取远程房间基础配置
9. 创建所有群的定时任务
10. 启动 Flask/gevent webhook 服务

说明它设计上希望做到：

- 程序重启后能恢复
- 配置尽量远程统一管理
- 业务尽量自动运转

# 六、核心业务机制：麦序状态机

这套系统最核心的不是“发消息”，而是**按小时档运行的状态机**。

## 6.1 每个房间按“小时档”工作

程序会按每个房间的配置，在某个分钟点触发 `task(room_wxid)`，相当于开始一档新的麦序。

每档会处理：

- 当前日期 / 当前小时
- 当前主持人
- 当前房间公告
- 扣排关键词 / 防脚本题目
- 榜单初始化
- 截止与补排规则

## 6.2 每档大致分为三个阶段

### 阶段 A：发档 / 开榜

机器人会在群里发一条开档消息，内容通常包含：

- 当前时段
- 主持人
- 公告
- 扣排方式 / 题目 / 验证码

然后打开以下状态：

- `isOpen = True`
- `isDetectKou = True`

### 阶段 B：手速 / 抢榜期

用户通过群消息发送关键词“扣排”，进入榜单。

这一阶段：

- 手速类任务值通常为 0
- 普通任务有不同 `value`
- 排序由 value + 时间决定
- 可发生顶榜、买8、插队、置顶等行为

### 阶段 C：截止 / 补排期

当达到截止时间或榜单满员时，进入截止逻辑：

- 如果榜满：`isStop = True`
- 如果榜没满：`isCanAdd = True`
- 动态加入“补”作为新的 0 值关键词
- 允许在一定时间内补排

随后发出：

- 手速榜
- 截止榜
- 打卡记录
- 统计上传

# 八、排麦规则机制

## 8.1 specialKeyJson：规则核心

`specialKeyJson` 是整个排麦系统的规则表，形如：

```json
{
  "手速": 0,
  "任务A": 10,
  "任务B": 20,
  "买8": 88
}
```

作用：

- 识别哪些消息算有效扣排
- 决定任务值大小
- 决定谁能顶掉谁
- 决定是否属于补排 / 特权 / 买8 / 置顶等

## 8.2 排序机制

虽然排序函数在其他模块，但从调用方式可判断：

- 同一用户重复扣排会去重，只保留最佳条目
- 排序核心依据是 `value`
- 同 value 再按时间先后
- 使用 `sort_and_no_repeat()` 做去重与排序

## 8.3 二阶段起排规则

如果设置了：

- `orderAddMinutes`
- `orderAddMinValue`

那么在某个时间点之后：

- 低于最低任务值的内容不允许进入榜单
- 只允许更大的任务继续补榜

这是典型的“前期手速，后期看任务值”的机制。

## 8.4 连排机制

如果开启 `isLianPai`：

- 上一档排过的人，本档必须按“连”规则续排
- 没排过的人不能直接扣“连”

说明程序支持**跨档业务规则**，不是每档完全独立。

# 十八、主要问题与风险

## 18.1 根本风险：底层依赖个人微信 Hook

无论功能多强，只要底座是：

- 个人微信号
- Hook / 注入 / 非官方客户端

那就天然存在：

- 封号风险
- 掉线风险
- 协议变更风险
- 微信风控升级风险

这不是代码能彻底解决的问题，是接入方式的问题。

## 18.2 单文件巨石

核心逻辑堆在 `deal_fun()` 和 `task()` 中，导致：

- 可维护性差
- 测试困难
- 迭代容易互相影响
- 接手成本高

## 18.3 并发与锁设计风险

虽然程序大量使用线程，但：

- `deal_fun()` 持有全局 `RLock`
- 锁内又执行网络请求、sleep、消息发送
- 而定时任务 `task()` 又不完全受同一套锁保护
- 全局状态被多个线程和调度器共同读写

容易出现：

- 阻塞
- 竞态
- 状态错乱

## 18.4 全局状态耦合严重

`all_configs_arr` 同时承担：

- 配置缓存
- 实时状态
- 临时缓存
- 业务上下文

后期会越来越难维护。

## 18.5 接口安全感较弱

Webhook 入口看不到强校验逻辑，若部署不当，存在风险。

## 18.6 自动更新机制风险

支持远程下载更新程序并重启，如果更新源受污染，风险会比较高。

# 二十、迁移目标：从“微信 Hook 版”迁移到“小程序 + 独立麦序服务”

下面给出一一映射的重构方案。

目标架构：

- **用户侧：微信小程序**
- **主持侧：Web 后台 / 小程序主持模式**
- **核心：独立麦序服务**
- **触达：微信官方通知能力**
- **存储：Redis + MySQL/PostgreSQL**
- **可选实时房间：RTC 服务**

核心原则：

> 微信不再是“执行器”，而是“入口 + 提醒器”；业务状态机回收到你自己的服务里。

# 二十八、最终结论

这份微信麦序机器人代码，核心业务设计其实已经很成熟：

- 它不是只会“排队”
- 而是整合了语音厅运营中的一整套玩法、规则、报表和管理动作

真正的问题不在业务，而在底座：

- 个人微信 Hook 决定了它始终有封号与稳定性风险
- 单体脚本 + 全局状态决定了它很难继续规模化演进

如果要走长期路线，最合理的升级方式不是继续优化 Hook，而是：

> **保留业务规则与状态机，重做接入层与运行架构。**

最推荐的方向是：

- **小程序承接用户交互**
- **独立麦序服务承接状态机与榜单**
- **微信只做触达和通知**

这条路的价值在于：

- 用户习惯能延续
- 核心资产回到自己手里
- 风险显著下降
- 后续接 Web / App / 其他入口都更容易
